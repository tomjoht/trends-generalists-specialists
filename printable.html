Why trends intrigue us

* in metrics, trends always popular * trends post this year almost went viral * focus on specialists and generalists

Argument overview

  • Technology is getting simpler on the front-end for end-users
  • But the code underneath is becoming increasingly specialized/complex
  • Tech writers are generalists, not specialists
  • To provide value in specialist contexts, tech writers must exploit the gaps
  • These gaps are (1) doc tools/processes, (2) understanding user feedback/experiences, and (3) information usability
* the direction I'm heading here * focusing on shaping forces in our industry * my views are shaped by my experiences (possibly US-centric), so please give input in surveys

2. Gaps in user feedback/experience

Identifying where the UX problems are

* "Why API Docs Fail" in IEEE software journal * main problems involve content -- incomplete, inaccurate, ambiguous, etc. * problem is that people who can fix it are too busy

Surface problems to the right teams

So, how can we improve API documentation if the only people who can accomplish this task are too busy to do it...? One potential way [is to] reduce as much of the administrative overhead of documentation writing as possible, letting experts focus exclusively on the value-producing part of the task. ... a main challenge for evolving API documentation is identifying where a document needs to be updated. — Uddin and Robillard
* you identify what's wrong * allows engineer to focus efficiently on correcting the problem * tech writer is broker between UX and engineers

Doc feedback buttons

* implement clear mechanism to report issues * very small amount of feedback (6 per 30,000 page views) * still valuable as mechanism for feedback. * should be seen as one of many channels

Doc feedback form

* how to elicit feedback not always straightforward * my approach: start simple, then elicit some response * get a lot of spam still

Surveys at select milestone events

* more targeted and useful * info is higher level, like "support is unresponsive" * kind of tedious and difficult in GDPR era

Summaries of weekly issues resolved

* Most impact internally * Particularly relevant in large orgs where no one knows who does what * Strengthening relationships, showing what we do week over week * Stuff it all into a short, readable email

Leveraging information flow through docs

* Most successful tweet ever * Docs is center of organization - line through which everyone flows * Idea is that you should pull info from all these groups * Feedback should flow * Connect SIM summaries with these "sentries"

3. Gaps in Information usability

* what is "information usability" - what designers do but with information * series of essays called Simplifying Complexity * selected 7 but there's no limit

p1: Give users a map

First we split complex process into chunks

* first principle of reducing complexity is chunking * but then you end up with a ton of chunks * could be many pages or topics in an installation topic

Maps bring the pieces together

* simple linear map created with CSS * easy but helps provide context * clear start to end - works well with certain processes (setup, submission, etc.)

p2: Make info discoverable as needed

Complex systems contain feedback loops

Each new piece of data the user uncovers affects the path taken and the eventual outcome. … it does not lend itself to being performed with a defined set of tasks nor can those tasks be performed in a fixed order.
— Michael Albers, Content and Complexity
* definition of a complex system is feedback loops, nonlinearity * can't really draw maps because too much branching, conditions, non-linearity * each step of way determined by previous step * example of describing buoys in water

Sintering as an analogy

* discrete pieces of info coming together to form coherent structure * think chatbots * promise of XML: right info at right time, place, language, device, user, etc. * more of a fun idea than a reality, since info chunks unwieldy/generic * plays out in SEO, discoverability. * google searches are cintering in action * your attempt to test SEO of topics

p3: Ensure harmony across all docs

Systems developed independently but interact together

Specialization is required in order to understand more and more about the intricate systems around us.... But at the same time, the systems we are building ... are not only intricate and complicated, but also stitch together field after field.... The design of driverless cars is a good example, requiring collaboration among ... software, lasers, automotive engineering, digital mapping, and more. — Samuel Arbseon
* complexity is due to interactions of indepently designed systems * APIs by definition stitch different systems together * when problems happen (toyota acceleration), people scramble through code

Looking for fit across the docs, blogs, forums

* engineers myopic about contributions * example of audio focus content * crunching through content like a grad level course

p4: Reduce and distill to its essence

Compress into smaller, more consumable units

* opposite activity as integration * no one has time to read lengthy articles * need immediate knowledge about whether direction is right * requires poet's astuteness and language command

Article summaries

* most useful -- Jakob Nielsen example * bake into your layout -- my blog as example * first 2 sentences of topic most important

Quick reference guides

* compress entirety into 1-2 page glance * allows users to see the whole * sidebar TOC useful for same purpose: see the whole

p5: Conform to genre expectations

Schema theory

By catering my design to meeting your experiences, I make these items easier for you to use in that context. — Kirk St. Amant
* scenario of checking into hotel * uses expect certain patterns, and this allows unconscious action * follow conventions of genre for successful discourse

Four Components Model

* Hans van der Meij poured through lots of doc * common sections users need/expect in docs * genre conventions

Narrative paradigm

* similar to four components model * i try to shape everything as a story * story almost always works * people too easily reduce story to dramatic fiction * nonfiction degree -- tension in ideas, the essay format

Structuring information around the customer journey

* competitive analysis project * all this information, how to structure/organize it? * journey just makes sense, even if semi-fictitious

p6: Reduce language complexity

I came across a set of API resources for managing a DEG the other day. You could add, update, delete and get DEGs. You can also pull analytics, history, and other elements of a DEG. I spent about 10-15 minutes looking around their developer portal, documentation, and even Googling, but never could figure out what a DEG was. Nowhere in their documentation did they ever tell consumers what a DEG was, you just had to be in the know I guess. — Kin Lane
* blind to acronyms, curse of knowledge * half of acrolinx setup is customizing your dictionary * particularly hard to distinguish familiar/unfamiliar terms with tech audience

[Not] Recognizing familiar terms

* example of "binary" with app submission * read competitor's documentation to broaden horizons * can't say if term is right if remain in your corporate bubble

Background knowledge sections

* Amazon's doc piggybacks on Android * help users ramp up to needed level * fits in with prerequisites model

p7: Iterative design of docs

Docs as a "theory" to test

In a sense, many things on a software development project are theories, or more accurately, assertions that need to be evaluated. ... Just because some stakeholders ... say a requirement is valid does not mean that they are correct. We need to evaluate even the requirements to determine whether they define the right solution to the problem at hand. — Spence and Bitner
* agile is way to be successful with complex projects * need iterative design b/c complex doc is hard to get right on first try * "document late" doesn't understand point of agile

Upon initial publication, begin iterative cycles

* after first publication, then you start iterating * doc is never done -- like Day 1 of docs * problem is bandwidth for continual iteration
